# 51LIS 低空智能监管系统 — 系统设计文档

> **版本**：v1.0  
> **代号**：LIS2 (Low-altitude Intelligent Supervision)  
> **定位**：面向多运营商、多无人机的低空飞行计划管理、时空冲突检测与基于 LLM 的智能协商解脱平台。

---

## 1. 项目概述

### 1.1 目标场景
在城市级低空空域中，多个无人机运营商会同时提交飞行计划。由于空域资源有限、禁飞区动态发布、各运营商对时间/高度/任务的柔性约束不同，存在**时空冲突**与**违反空域规则**的风险。本系统提供：

- 飞行计划的全生命周期管理（CRUD + 自动航路规划）；
- 禁飞区（GeoJSON 多边形 + 高度区间）管理；
- 计划之间的两两冲突 / 全局冲突检测；
- 基于 DeepSeek LLM + Function Calling 的**冲突协商 Agent**，自动调时间 / 调高度 / 设避让区 / 重规划航路；
- 基于 WDP 三维实景引擎的态势可视化。

| `interruptible` | INTEGER (0/1) | **任务可中断** |

- `geometry`：GeoJSON `Polygon` 序列化字符串。
- `min_alt` / `max_alt`：高度区间，路径规划时仅约束此高度内的网格。
- `start_time` / `end_time`：动态禁飞窗口（当前 routing 实现 **跳过时间限定的禁飞区**，TODO 待补强）。

#### conflict_events
| 字段 | 说明 |
|------|------|
| `id`、`type`(默认 `time_space`)、`severity` (`high`/`medium`/`low`) |
| `involved_plans` | 参与计划 ID 数组的 JSON 字符串 |
| `time_window_start/end` | 冲突时间窗口（多段合并后的总区间） |
| `conflict_point_lon/lat`、`conflict_alt`、`distance` | 最近点信息 |
| `status` | `unresolved` / `in_progress` / `resolved` |
| `parent_event_id` | 预留，用于子冲突追溯 |

#### conflict_details
- 一对多展开 `conflict_events` ↔ `flight_plans`，记录每条计划的涉及时间区间。

#### agent_sessions / session_messages
- 一个冲突可创建多个 Agent 会话；每轮 user / assistant / tool 消息持久化到 `session_messages`。

> **持久化策略**：`sql.js` 是 WASM 版 SQLite，数据库整体加载到内存。每次 `run()` 后会立刻把整库 `export()` 后 `fs.writeFileSync` 回磁盘。**性能取舍**：写多读少场景会显著放大 IO，目前数据规模小可接受；后续如需高并发可替换为 `better-sqlite3`。

### 3.4 航路规划（routingService）

源于 [server/services/routingService.js](server/services/routingService.js#L1)。规划链路是 **A\* 网格搜索 + 直线兜底 + 可视图绕飞**，并对结果做平滑与高度包络：

1. **坐标系转换**：以起点为参考原点，将经纬度投影为本地米制平面 `(x, y)`。
2. **栅格构建**：分辨率 `GRID_RESOLUTION = 10m`；边界在起终点外扩 500m。
3. **障碍物注入**：
   - `loadNoFlyZonesAsGrid`：将永久禁飞区栅格化标记为 `'nofly'`（**跳过任何有 `start_time`/`end_time` 的时间限定禁飞区**）。
   - `addAvoidanceZones`：支持两种形式
     - 圆柱形：`{ center_lon, center_lat, altitude, radius }`，高度上下各扩 50m；
     - 多边形：GeoJSON Polygon + `min_alt`/`max_alt`。
4. **A\***（[`astar3D`](server/services/routingService.js#L131)）：
   - 26 邻接（含上下层移动），垂直步长惩罚 `VERTICAL_PENALTY = 4`；
   - 限制爬升率 `5 m/s`，约束 `|dz| ≤ ceil(5*60/10)`；
   - 启发式：水平欧氏 + 垂直曼哈顿带惩罚；
   - 上限 `MAX_ITERATIONS = 200000`。
5. **路径平滑** `smoothPath`：贪心式 line-of-sight，去除冗余拐点。
6. **直线兜底** `generateDirectWaypoints`：当 A\* 未找到路径或没有任何障碍物时，使用 20 等分直线，并对 Polygon 类避让区使用**可视图（visibility graph）+ Dijkstra** 计算最短绕飞路径，最后对单点做"推出多边形"修正。
7. **高度包络** `applyAltitudeEnvelope`：起飞段 / 巡航段 / 下降段三段，巡航最低高度 `MIN_FLIGHT_ALT = 200m`。
8. **缓存**：`SIMPLE_PATHS` 以 `JSON.stringify({start,end,avoidanceZones})` 为键，重复请求直接命中（**注意**：当前缓存无失效策略，禁飞区数据变化后不会自动失效）。

输出：`{ type: 'LineString', coordinates: [[lon, lat, alt], ...] }`，与 GeoJSON 兼容，可直接喂给 WDP 引擎渲染。

### 3.5 冲突检测（conflictService）

源于 [server/services/conflictService.js](server/services/conflictService.js#L1)。

#### 算法核心
1. **航路插值** `interpolateWaypoints`：按 **1 秒采样间隔** 在 `start_time → end_time` 之间均匀采样三维位置（经纬度按经度纬度米制简化转换，高度直接相减）。
2. **两两比对**：仅比较时间差 `|t1 - t2| ≤ 5s` 的样本，3D 距离阈值 `threshold = 15m`。
3. **冲突窗口合并**：相邻冲突样本时间间隔 ≤ 10s 视为同一窗口，合并为 `time_windows: [{start, end}]`。
4. **严重度判定**：基于最小间距 minDist
   - `< 3m` → `high`
   - `< 5m` → `medium`
   - 其他 → `low`
5. **全局检测去重**：读取所有未解决冲突的 `involved_plans` 排序拼接作为 pair key，避免重复入库。

> **复杂度**：当前 `O(n_samples1 × n_samples2)`，长航段会很慢；后续可改 KD-Tree 或时间分桶。

### 3.6 智能 Agent（agentService）

源于 [server/services/agentService.js](server/services/agentService.js#L1)。

#### LLM 配置
- 模型：`deepseek-chat`
- API：通过环境变量 `DEEPSEEK_API_KEY` / `DEEPSEEK_API_URL`（默认 `https://api.deepseek.com/chat/completions`）
- 协议：OpenAI 兼容的 `tools` + `tool_choice: 'auto'`

#### SYSTEM_PROMPT 关键约束
> 优先级：**调时间 > 调高度 > 绕飞避让**；每次调整后必须重新检测冲突；最终给出清晰建议。

#### 工具集（Function Calling）

| 工具名 | 作用 | 关键参数 |
|--------|------|---------|
| `query_conflict_detail` | 拉取冲突详情 + 涉及计划 | `conflict_id` |
| `query_plan_detail` | 拉取单计划详情 | `plan_id` |
| `adjust_time` | **校验 `time_flexible`** 后偏移起降时间 | `plan_id`, `offset_minutes` |
| `adjust_altitude` | **校验 `alt_flexible`** 后改 start_alt / end_alt | `plan_id`, `new_altitude` |
| `set_avoidance_zone` | 返回一个圆柱形避让区描述（不入库，供后续 replan 用） | center/altitude/radius |
| `replan_route` | 调用 routingService 重新规划并落库 | `plan_id`, `avoidance_zones[]` |
| `detect_conflict_for_plans` | 调整后重检 | `plan_ids[]` |
| `confirm_solution` | 提交冲突 `status = resolved` | `conflict_id`, `changes[]` |

#### 多轮调度流程（`runAgentRound`）

```
1. 持久化用户消息
2. 拼装 messages:
   [system prompt, system(当前冲突上下文 JSON), ...history]
3. while loopCount < MAX_LOOPS(10):
     resp = callLLM(messages)
     if resp.finish_reason == 'tool_calls':
         对每个 tool_call:
             - wsSend(tool_call) → 前端实时展示
             - 持久化到 session_messages (role=assistant, content=JSON)
             - 本地执行 executeToolCall
             - wsSend(tool_result) + 持久化 (role=tool)
             - messages 追加 assistant(tool_calls) + tool(result)
         continueLoop = true
     else:
         wsSend(text) + 持久化
         wsSend(confirmation_required)   # 等待前端 confirm_solution
         break
```

#### WebSocket 通道（[server/routes/ws.js](server/routes/ws.js#L1)）

- URL：`/ws/agent?sessionId=<uuid>`
- 服务端 → 客户端消息类型：`text` / `tool_call` / `tool_result` / `confirmation_required` / `session_end` / `error`
- 客户端 → 服务端消息类型：`user_message` / `confirm_solution` (`accept: boolean`) / `cancel_session`
- 连接建立后，若会话已绑定 `conflict_event_id`，**自动**触发一轮"请分析此冲突并提出解决方案"。

### 4.4 3D 引擎与覆盖物

#### 引擎初始化（[useWdpEngine.js](client/src/composables/useWdpEngine.js#L1)）
- **单例**：模块级 `_App / _initPromise / _sceneReady / _loadingProgress`。
- 依赖环境变量 `VITE_WDP_URL` / `VITE_WDP_ORDER`（未配置时跳过 3D 渲染，进入 fallback）。
- 监听 `OnWdpSceneIsReady` 事件，把 `progress 0~100` 暴露给 `WdpViewer` 的加载蒙层。
- 场景就绪后由调用方传入 `onSceneReady(App)` 钩子完成渲染模式、光照等初始化（见 `WdpViewer.vue` 中 `SetRendererMode('full', [3840, 2160])`、`SetSkylightTime('18:30')`）。

- `drawConflict` — 冲突点高亮；
- `removePlanEntity` / `removeZoneEntity` / `removeConflictEntity` — 通过 `customId` 检索后删除。

#### 覆盖物管理（[useOverlayManager.js](client/src/composables/useOverlayManager.js#L1)）
- `initOverlays()`：场景 ready 后批量绘制全部数据；
- `addPlan / removePlan`、`addZone / removeZone`、`addConflict / removeConflict`：单点增删；
- `onPlansChange / onNoFlyChange / onConflictsChange`：被 `WdpViewer` 中 `watch(deep)` 调用做 diff 增量更新，避免全量重绘。

### 4.5 主题与视觉

参见 [UI设计规范.md](UI设计规范.md#L1) — 黑绿科技风：
- 主背景 `#0A0F0F`、强调色 `#78BE2D`；
- naive-ui 通过 `themeOverrides` 注入主色 / 圆角 / 弹层底色；
- 全局 CSS 变量定义在 `assets/global.css`，并被各组件复用（如 `--green-glow`、`--shadow-glow`、`--radius-md`）。

## 5. 端到端关键流程

### 5.1 新建计划（含自动航路）
```
PlanPanel 表单 → plansStore.createPlan(data)
  → POST /api/plans
      → planService.createPlan (若 waypoints 缺失则先 routingService.planRouteDirect)
      → INSERT flight_plans, saveDb()
  → 前端 fetchPlans() → plansStore.plans 更新
  → WdpViewer watch(plansStore.plans) → useOverlayManager.onPlansChange
      → 调用 drawPlanPath / drawPlanDeparture / drawPlanArrival
```

### 5.2 全局冲突检测
```
ConflictPanel "全局检测" → conflictStore.detectGlobal()
  → GET /api/conflicts/detect
      → conflictService.detectGlobal
          1. 读取所有 active 计划
          2. 读取所有未解决冲突 → 构造 pair-key 集合用于去重
          3. 两两 detectConflict（采样 + 3D 距离）
          4. 命中则 INSERT conflict_events + conflict_details
      → 返回新增冲突列表
  → 前端 fetchConflicts() → ConflictPanel 列表渲染
  → WdpViewer 增量绘制冲突点
```

### 5.3 智能冲突解脱
```
ConflictPanel "智能解脱" → router.push(`/agent/${conflictId}`)
AgentView onMounted:
  → api.createSession({ conflictEventId })  → 拿到 sessionId
  → new WebSocket(`/ws/agent?sessionId=...`)
ws.onopen:
  服务端自动 wsSend(text "会话已建立...") + 触发 runAgentRound(初始请求)
  
runAgentRound 循环：
  LLM → tool_call (e.g. query_conflict_detail) → 本地执行 → 结果回灌
  LLM → tool_call (adjust_time / replan_route ...) → 落库
  LLM → tool_call (detect_conflict_for_plans) → 验证
  LLM → text + confirmation_required

用户点击「接受」→ ws.send({ type:'confirm_solution', accept:true })
  → handleConfirm → conflict.status = 'resolved' → ws.send(session_end)
  → 前端 emit('resolved') → 显示 ✅ 蒙层
```

## 6. 安全与运维

### 6.1 当前实现
- CORS 开放（`app.use(cors())`）— 仅适用于内网原型。
- 请求体限制 `10mb`（兼容大 `waypoints` JSON）。
- `DEEPSEEK_API_KEY` 通过 `.env`，未硬编码。
- WebSocket 不要求鉴权，**仅校验 sessionId 存在**。

### 6.2 已知风险 / TODO
1. **无身份认证**：API 与 WS 完全开放，生产部署前需补 JWT / OAuth。
2. **sql.js 全库回写**：写并发与大数据量下成为瓶颈，建议替换为 `better-sqlite3`。
3. **routingService 缓存无失效**：禁飞区变更后旧航路缓存未清理。
4. **时间限定禁飞区被规划忽略**：`if (startTime || endTime) continue`，需补全时间窗与计划航段的相交判断。
5. **冲突检测 O(N²) 全采样**：长航段或多计划场景需做时间窗预筛 / 空间索引。
6. **Agent 工具执行无事务**：`adjust_time` / `replan_route` 失败时不会回滚，用户拒绝方案后**修改已落库**（仅 `conflict.status` 不变为 resolved，但 plan 已 modified）— 建议为 Agent 会话引入"草稿态"或事务快照。
7. **LLM 调用无重试 / 限流**：网络抖动会直接终止会话。
